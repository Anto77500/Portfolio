Projet 03 · Intégrité & Sauvegardes

Gestion de Base de Données Relationnelle

Sécurisation, cloisonnement des privilèges SQL et automatisation des sauvegardes sur un serveur MariaDB.

ContexteStart-up MedicaClick
SGBDRMariaDB / MySQL Linux
SécuritéScript Shell / Cron Job
1. Contexte & Problématique

La start-up MedicaClick développe une plateforme de prise de rendez-vous médicaux et de gestion d'approvisionnement pour les structures de santé. L'application Web s'appuie sur une base de données relationnelle stockant des données d'utilisateurs et des informations organisationnelles sensibles.

Problématique identifiée : Initialement, l'application se connectait à la base de données en utilisant le compte d'administration global `root`. En cas de faille de sécurité ou d'injection SQL sur l'interface Web, un attaquant pouvait obtenir le contrôle total de l'ensemble du serveur de données, supprimer des tables entières ou altérer le système de fichiers sous-jacent. De surcroît, aucun mécanisme de sauvegarde régulier n'était planifié, menaçant la continuité d'activité en cas de panne matérielle.

2. Objectifs Techniques

Afin de s'aligner sur les bonnes pratiques de sécurité de l'ANSSI, la réalisation technique doit valider les critères suivants :

  • Durcissement du SGBDR : Éliminer l'ensemble des vulnérabilités d'installation d'origine (comptes invités, bases de tests publiques).
  • Principe du moindre privilège : Restreindre l'accès applicatif Web à un utilisateur dédié, doté uniquement des permissions d'exécution nécessaires (SELECT, INSERT, UPDATE) en excluant les droits destructifs.
  • Automatisation de la résilience : Mettre en œuvre une routine autonome de sauvegarde logique nocturne avec horodatage pour archiver les données sans intervention humaine.

4. Réalisation Technique

Étape 4.1 : Durcissement initial de l'instance MariaDB

Installation du moteur SQL puis sécurisation de l'environnement d'exécution via le script d'initialisation recommandé :

apt install mariadb-server -y && mysql_secure_installation

Cette étape permet de configurer un mot de passe `root` fort, de désactiver les connexions anonymes et de bloquer l'accès distant à ce compte d'administration.

Étape 4.2 : Cloisonnement SQL et application des privilèges

Création de la base de données dédiée à MedicaClick et assignation des privilèges stricts à l'utilisateur applicatif local :

CREATE DATABASE bts_infrastructure;
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'MotDePasseFortEtSûr123!';
GRANT SELECT, INSERT, UPDATE ON bts_infrastructure.* TO 'app_user'@'localhost';
FLUSH PRIVILEGES;

Étape 4.3 : Automatisation des sauvegardes régulières

Rédaction d'un script d'automatisation shell (Bash) utilisant l'outil d'extraction logique mysqldump :

#!/bin/bash
DATE=$(date +%Y-%m-%d)
mysqldump -u root -p'password' bts_infrastructure > /backup/db_backup_$DATE.sql

5. Bilan & Compétences

Bilan opérationnel : La plateforme MedicaClick dispose désormais d'un socle de données sécurisé et résilient. Une faille applicative Web ne permet plus de compromettre le serveur SGBDR complet grâce au cloisonnement utilisateur. Les archives SQL générées quotidiennement garantissent la non-perte de données en production.
